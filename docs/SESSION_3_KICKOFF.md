## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (разделы 5, 7.1, 7.6). Сначала план, потом код.

### 1. Интеграция Telegram WebApp SDK

- Подключи официальный SDK Telegram WebApp. На монтировании приложения вызови
  `WebApp.ready()` и `WebApp.expand()`.
- Читай сырую строку `window.Telegram.WebApp.initData` и шли её в заголовке
  `Authorization: tma <initData>` на КАЖДЫЙ запрос к API (через центральный
  api-клиент). НИКОГДА не используй `initDataUnsafe` для логики — только для
  косметики, и то осторожно.
- Уважай тему Telegram: привяжись к `themeParams` / `colorScheme` (светлая/тёмная),
  не хардкодь цвета.

### 2. Dev-режим без Telegram (важный камень)

Открытый в обычном браузере Vite не имеет `window.Telegram` → `initData` пустая →
401. Чтобы итерировать UI вне Telegram:
- если `window.Telegram` отсутствует И `import.meta.env.DEV`, подставляй мок
  валидной `initData` из env-переменной `VITE_DEV_INIT_DATA`;
- сгенерируй этот мок тем же алгоритмом подписи, что в тестах Сессии 1
  (dev-токен, свежий `auth_date`), для `telegram_id` засиженного руководителя;
- жёстко исключи этот путь из прод-сборки (`import.meta.env.DEV`).

### 3. Центральный API-клиент

Обёртка, которая вешает заголовок авторизации и переводит ошибки в понятные
локализованные сообщения:
- 401 → «Откройте приложение из Telegram»;
- 403 (нет в базе) → «Вы ещё не добавлены руководителем»;
- 403 (роль) → не показывать интерфейс руководителя.

### 4. Разделение по ролям (по `role` из `/api/me`)

- **worker**: переключатель + «мой день» (визиты, часы, «на смене с HH:MM»);
- **manager**: сводка за день (кто отметился, времена визитов, суммарные часы)
  через `/api/attendance/summary`.
- Проверка роли на клиенте — ТОЛЬКО для отрисовки. Безопасность уже на бэкенде
  (manager-эндпоинты сами отдают 403 рабочему).

### 5. Переключатель «Пришёл/Ушёл» + debounce

- Используй нативную `MainButton` Telegram (подпись меняется по состоянию:
  «Пришёл» при закрытом, «Ушёл» при открытом визите); дёргает
  `POST /api/attendance/toggle`.
- БЛОКИРУЙ кнопку на время запроса (debounce) — клиентская половина защиты от
  двойного тапа. Показывай состояние загрузки, разблокируй по ответу.
- После ответа обнови состояние («на смене с HH:MM» / часы за сегодня).

### 6. Локализация (vue-i18n, ru/uz)

- Настрой `vue-i18n` с файлами ru/uz. Переводи ТОЛЬКО подписи интерфейса и
  сообщения. Структуру сделай так, чтобы модули Этапов 2–3 её переиспользовали.
- Язык: `employee.language` из `/api/me`; fallback — Telegram `language_code`;
  затем `ru`.
- Время отображай в `Asia/Tashkent` (часовой пояс предприятия), НЕ в локали
  устройства.

### 7. Проверка (фронт — по нашему правилу, легко)

- `make check` должен проходить (eslint + сборка `vite build`).
- Несущая UX-проверка: кнопка заблокирована во время запроса (один тест на это
  достаточен, если используешь vitest; иначе ручная проверка).
- Полная проверка — вручную в Telegram через тоннель.

### Definition of Done (Сессия 3)

- В Telegram (через тоннель) рабочий видит кнопку, отмечает приход/уход; двойной
  тап безопасен (кнопка блокируется). Руководитель видит сводку за день.
- Экран уважает светлую/тёмную тему Telegram.
- Язык переключается ru/uz по профилю.
- В dev мок `initData` позволяет открывать приложение в обычном браузере.
- `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.

Это замыкает Этап 1 ТЗ (тонкая вертикаль посещений работает насквозь). UI сверки
и ручной правки руководителем — Сессия 3b. Дальше — Этап 2 (Заказы).
