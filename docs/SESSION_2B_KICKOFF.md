## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (раздел 7.1, пункты 7 и 8). Сначала план,
потом код. Останавливайся на ревью перед любой миграцией и перед коммитом.

Все эндпоинты этой сессии — ТОЛЬКО для роли `manager` (через `get_current_employee`
+ проверка роли).

### 1. Сверка по камере (тз 7.1.7)

`POST /api/attendance/{id}/verify` с телом `{status: "confirmed" | "discrepancy"}`:
- проставляет `verification_status`, `verified_by` = текущий руководитель,
  `verified_at` = now();
- повторная сверка разрешена (можно сменить `confirmed` ↔ `discrepancy`),
  при этом `verified_by`/`verified_at` обновляются.

### 2. Статистика расхождений (тз 7.1.7)

`GET /api/attendance/discrepancies?from=...&to=...` — по сотрудникам количество
записей со статусом `discrepancy` за период. Только чтение, manager.

### 3. Ручная правка записи (тз 7.1.8)

`PATCH /api/attendance/{id}` — правка `check_in_at`, `check_out_at`, `note`
(типичный кейс — закрыть «забытый» визит, проставив `check_out_at`).

ВАЖНО, реши в плане ДО кода (меняет схему):
как фиксировать провенанс правки (кто/когда/что менял). Предложи варианты —
(a) только `note` + `updated_at`; (b) поля `edited_by`/`edited_at`; (c) мини-история
`attendance_adjustment` — и ОСТАНОВИСЬ на моём решении. Рекомендация по умолчанию:
вариант (b). После решения — отдельная миграция на ревью.

### 4. Валидация правки

- Добавь CHECK на уровне БД: `check_out_at IS NULL OR check_out_at > check_in_at`
  (отдельная миграция; место именно здесь — переключатель это нарушить не может,
  ручная правка может). В сервисе нарушение → 400 с понятным сообщением.
- Если правка делает визит снова открытым (`check_out_at = NULL`), а у сотрудника
  уже есть другой открытый визит → частичный индекс даст `IntegrityError`. Поймать
  и отдать 409 с понятным сообщением, не 500.
- При изменении `check_in_at` пересчитай `work_date` в `SHOP_TZ` (`Asia/Tashkent`).
- Закрытие открытого визита (проставление `check_out_at`) — штатный кейс, не ошибка.

### 5. Несущие тесты (только эти, против настоящего Postgres, через auth-фикстуру)

- `verify` ставит статус + `verified_by`/`verified_at`; рабочему → 403;
- повторная `verify` меняет статус и обновляет `verified_by`/`verified_at`;
- правка с `check_out_at < check_in_at` → 400 (и CHECK на уровне БД срабатывает);
- правка времени корректно пересчитывает часы за день;
- reopen визита при уже открытом другом → 409 (не 500);
- сдвиг `check_in_at` через полночь меняет `work_date` по `Asia/Tashkent`;
- `discrepancies` корректно считает за период; рабочему → 403.

### Definition of Done (Сессия 2b)

- Руководитель помечает записи `confirmed`/`discrepancy`; видит статистику
  расхождений; правит записи руками; провенанс правки фиксируется по выбранному
  варианту.
- Миграции (CHECK + поля провенанса, если выбраны) применены и чисто
  откатываются; показаны на ревью до применения.
- `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.
