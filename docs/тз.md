# Техническое задание

## Система управления мебельным производством (Telegram-решение)

---

## 1. Общие сведения

### 1.1. Назначение
Кастомная система автоматизации для одного небольшого предприятия по производству мебели. Закрывает три задачи: контроль посещений сотрудников, управление заказами и базовый финансовый учёт с отчётностью.

### 1.2. Тип продукта
Решение строится **целиком в экосистеме Telegram**:
- **Telegram-бот** — единая точка входа: запускает Mini App для сотрудников, обслуживает клиентов в обычном чате, в перспективе рассылает уведомления.
- **Telegram Mini App** (веб-приложение внутри Telegram) — основной рабочий интерфейс для сотрудников и руководителя.

### 1.3. Ключевые принципы
- **Один продукт под одно предприятие.** Простые процессы, минимум избыточной функциональности.
- **Без барьера установки.** У всех пользователей уже есть Telegram; отдельное приложение ставить не нужно.
- **Единый бэкенд, несколько «лиц».** Вся логика и данные — на одном сервере; поверх него работают разные интерфейсы под разные роли.
- **Работа и с телефона, и с компьютера.** Тяжёлую работу (заказы, отчёты) руководитель может вести в Telegram Desktop на большом экране — это тот же Mini App.
- **Двуязычный интерфейс.** Бот и Mini App работают на двух языках (русский и узбекский); язык выбирается пользователем.

---

## 2. Цели проекта
1. Дать руководителю прозрачный учёт посещаемости с возможностью сверки по камерам видеонаблюдения.
2. Вести заказы от создания до готовности, хранить прикреплённые файлы (чертежи, фото) и показывать клиенту статус его заказа.
3. Вести учёт денег по заказам (полученные оплаты и расходы) и формировать базовые месячные отчёты.

---

## 3. Архитектура решения

### 3.1. Компоненты
| Компонент | Назначение |
|---|---|
| Backend (API) | Бизнес-логика, доступ к данным, валидация, отчёты. Обслуживает Mini App и бота. |
| Telegram-бот | Запуск Mini App, обработка запросов клиентов, уведомления. |
| Mini App (фронтенд) | Интерфейс сотрудников и руководителя. |
| База данных | Единое хранилище всех данных. |
| Файловое хранилище | Хранение файлов заказов и их миниатюр. |

### 3.2. Поверхности входа
1. **Mini App рабочего** — отметка посещений, просмотр своих заказов, смена статуса (если разрешено).
2. **Mini App руководителя** — полный доступ ко всем функциям всех модулей.
3. **Чат с ботом для клиента** — без авторизации; клиент по коду заказа получает его статус.

### 3.3. Принцип взаимодействия
Mini App обращается к бэкенду по HTTPS. Бот и Mini App используют один и тот же бэкенд и одну базу. Видеопотоки камер **в систему не интегрируются** — камеры остаются у руководителя отдельно, в системе фиксируется только результат сверки.

## 5. Пользователи и роли

### 5.1. Роли
- **Руководитель (manager)** — полный доступ ко всем модулям и справочникам.
- **Рабочий (worker)** — отмечает свои посещения; заказы по умолчанию только просматривает.
- **Клиент** — внешний пользователь без учётной записи; только просмотр статуса своего заказа через бота.

### 5.2. Матрица доступа

## 6. Авторизация и безопасность

### 6.1. Сотрудники
- Авторизация через **Telegram `initData`**: Mini App передаёт подписанные данные, бэкенд проверяет HMAC-подпись по токену бота и опознаёт сотрудника по `telegram_id`.
- Паролей и отдельного логина нет — личность подтверждает Telegram. Разграничение прав — по роли.

### 6.2. Клиенты
- **Без авторизации.** Данные о статусе несекретны.
- Идентификатор заказа для клиента — **неугадываемый случайный код** (например, `K7QX2M`), а не порядковый номер. Это исключает перебор и «слив» всего портфеля заказов.
- Поддержка deep-link: `t.me/<bot>?start=<код>` — клиент по ссылке сразу видит статус без ручного ввода.
- Рекомендуется лёгкое ограничение частоты запросов (rate limiting) на поиск по коду.

### 6.3. Транспорт
- **HTTPS обязателен** (требование Telegram к Mini App): нужен домен с SSL-сертификатом. На этапе разработки допустим туннель (например, ngrok).

## 7. Функциональные требования

### 7.1. Модуль «Посещения»

**Назначение.** Учёт фактически отработанных часов с возможностью выборочной сверки руководителем по камерам.

**Требования:**
1. Рабочий отмечает приход и уход. **За один день допускается несколько визитов** (рабочий может уходить и возвращаться).
2. Одна запись посещения = один визит (пара приход + уход).
3. Кнопка в Mini App работает как **переключатель**:
   - нет открытого визита → кнопка «Пришёл» (создаётся запись с временем прихода);
   - есть открытый визит → кнопка «Ушёл» (в эту запись проставляется время ухода).
4. У сотрудника **не может быть двух открытых визитов одновременно** (обеспечивается частичным уникальным индексом по `employee_id` при `check_out_at IS NULL`).
5. **Отработанные часы за день** = сумма длительностей всех визитов за этот день. За месяц — сумма по дням.
6. Руководитель видит сводку за день/период: кто отметился, времена визитов, суммарные часы.
7. **Сверка по камере:** руководитель выборочно сверяет факт присутствия и помечает запись — `confirmed` (сверено, ок) или `discrepancy` (расхождение); фиксируется, кто и когда сверял. Со временем накапливается статистика расхождений по сотрудникам.
8. Руководитель может **редактировать запись вручную** (например, если рабочий забыл отметить уход).

**Вне модуля (по согласованию):** перерывы/паузы внутри визита, графики и контроль опозданий, геолокация, учёт отпусков и больничных.

---

### 7.2. Модуль «Заказы»

**Назначение.** Ведение заказов от создания до готовности, хранение файлов и публикация статуса для клиента.

**Требования:**
1. **Руководитель** имеет полный доступ: создание, редактирование, смена статуса, загрузка файлов, прикрепление сотрудников, ведение расходов.
2. **Рабочие** по умолчанию имеют доступ только на чтение. Руководитель может выдать отдельным рабочим право **только на смену статуса** конкретного заказа.
3. **Прикрепление сотрудников:** к одному заказу можно прикрепить нескольких рабочих (связь «многие-ко-многим»). При прикреплении руководитель отмечает флаг «может менять статус» для нужных сотрудников (право действует в рамках этого заказа).
4. **Этапы производства** ведутся руководителем как **справочник** (см. 7.5). Этапы не зашиты в код: руководитель сам формирует список (например, «Раскрой», «Сборка», «Покраска») и их порядок. Текущий статус заказа — ссылка на этап из справочника.
5. **История смены статусов** фиксируется (какой этап, кто и когда установил, комментарий).
6. **Файлы заказа:** к заказу можно прикрепить несколько файлов (чертежи, картинки и т.д.). Подробности — раздел 8.
7. Поля заказа: код для клиента, внутренний номер, данные клиента, название и описание, текущий этап, срок готовности, сумма заказа, публичный комментарий для клиента.
8. **Расходы по заказу** (см. 7.4) ведёт руководитель прямо в карточке заказа.

---

### 7.3. Клиентский просмотр статуса заказа

**Назначение.** Дать клиенту возможность узнать состояние своего заказа через обычный чат с ботом, без авторизации.

**Требования:**
1. Клиент пишет боту **код заказа** (или переходит по deep-link) и получает текущий статус.
2. Если код не найден — понятное сообщение об ошибке.
3. Клиент видит **только безопасное подмножество данных**, формируемое функцией-сериализатором:
   - название и описание заказа;
   - текущий этап в виде прогресса «этап X из Y»;
   - срок готовности;
   - публичный комментарий руководителя.
4. Клиент **НЕ видит:** любые денежные показатели, файлы (чертежи), прикреплённых сотрудников, внутренние комментарии.
5. **На будущее (не в первой версии):** при первом обращении клиента сохраняется его `chat_id`, что позволит автоматически уведомлять клиента о смене статуса («Ваш заказ готов»).

---

### 7.4. Модуль «Финансы и отчётность»

**Назначение.** Простой денежный учёт по заказам и базовая отчётность. Ведёт только руководитель (отдельного бухгалтера нет).

**Требования:**
1. **Деньги привязаны к заказам.** По каждому заказу учитываются:
   - **Полученные от клиента средства** (авансы/оплаты, «заклады») — отдельные записи об оплате.
   - **Расходы по заказу.**
2. **Двухуровневая модель расхода:**
   - **Направление (уровень 1):** одно из двух —
     - `to_employee` — сумма, выданная конкретному сотруднику (подотчёт; обязательно указывается сотрудник);
     - `general` — расходники / прочие расходы (без привязки к сотруднику).
   - **Категория (уровень 2):** значение из **справочника категорий расходов** (см. 7.5), который ведёт руководитель.
3. **Общие (операционные) расходы предприятия.** Помимо расходов по заказам, ведётся учёт расходов, **не привязанных к конкретному заказу** (аренда, коммунальные услуги, связь и т.п.). Они используют тот же справочник категорий и хранятся в отдельной таблице (см. 9.11). В первой версии вносятся вручную руководителем.
4. **Отчётность.** На первом этапе — **месячные отчёты**, вычисляемые «на лету» (не хранятся):
   - доходы за месяц = сумма полученных оплат;
   - расходы за месяц = сумма расходов по заказам + операционные расходы, с разбивкой по категориям;
   - прибыль = доходы − (расходы по заказам + операционные расходы).
5. **Производные данные (на будущее):** поскольку у заказа есть сумма и связанные оплаты, остаток долга клиента (дебиторка) вычисляется автоматически.

---

### 7.5. Справочники

Несколько списков ведёт руководитель из своего кабинета. Они устроены одинаково и реализуются **единым переиспользуемым механизмом** «управление списком»:
- **Этапы производства** (`production_stage`) — имя + порядок следования + признак активности.
- **Категории расходов** (`expense_category`) — имя + признак активности.

Элементы справочников не удаляются жёстко, а помечаются как неактивные (чтобы не разрушать исторические данные).

---

### 7.6. Локализация (двуязычный интерфейс)

**Назначение.** Бот и Mini App доступны на двух языках — **русском и узбекском**.

**Требования:**
1. Переводится **только интерфейс**: подписи, кнопки, сообщения бота, шаблоны ответов клиенту. Реализуется через файлы переводов (`vue-i18n` для Mini App, механизм i18n в `aiogram` для бота); бизнес-логика от языка не зависит.
2. Значения справочников (названия этапов, категорий расходов) и пользовательский контент (названия и описания заказов, комментарии) **не переводятся** — отображаются так, как их ввёл руководитель.
3. **Язык сотрудника** хранится в его профиле (`employee.language`); по умолчанию берётся из Telegram (`language_code`), переключается вручную в настройках.
4. **Язык клиента:** бот определяет язык автоматически по данным Telegram либо предлагает выбор при первом обращении (`/start`).

**Вне модуля (на будущее):** двуязычные значения справочников; добавление третьего языка (архитектура переводов это допускает без переделки логики).

## 8. Работа с файлами

1. Файлы загружаются **через веб-интерфейс Mini App напрямую на бэкенд по HTTPS**, а не через Telegram-бота. Это снимает ограничения Telegram на размер файла у ботов и даёт полный контроль над типами и размерами.
2. **Хранилище:** на старте — файловая система сервера со структурой по заказам; метаданные — в базе. Работа с хранилищем выносится за **абстракцию (интерфейс)**, чтобы позже без переписывания логики перейти на S3-совместимое хранилище.
3. Для изображений (чертежи-картинки, фото) **генерируются миниатюры** (Pillow), чтобы список файлов не подгружал полноразмерные изображения.
4. CAD-файлы (DWG/DXF) превью не имеют — отображаются как скачиваемые по иконке.
5. Файлы — **строго внутренние**, клиенту через бота не показываются.

## 9. Модель данных

> Обозначения: PK — первичный ключ, FK — внешний ключ. Денежные суммы — `NUMERIC(12,2)`. Метки времени — `TIMESTAMPTZ`. Валюта единая (UZS).

### 9.1. `employee` — сотрудники
| Поле | Тип | Описание |
|---|---|---|
| id | PK | Идентификатор |
| telegram_id | BIGINT, UNIQUE | Telegram-идентификатор |
| full_name | TEXT | ФИО |
| role | ENUM(worker, manager) | Роль |
| language | ENUM(ru, uz) | Язык интерфейса (по умолчанию из Telegram) |
| position | TEXT, NULL | Должность |
| is_active | BOOLEAN | Активен |
| created_at | TIMESTAMPTZ | Дата создания |

### 9.2. `attendance_record` — посещения (1 запись = 1 визит)
| Поле | Тип | Описание |
|---|---|---|
| id | PK | Идентификатор |
| employee_id | FK → employee | Сотрудник |
| work_date | DATE | Дата визита |
| check_in_at | TIMESTAMPTZ | Время прихода |
| check_out_at | TIMESTAMPTZ, NULL | Время ухода (пусто = визит открыт) |
| verification_status | ENUM(not_checked, confirmed, discrepancy) | Результат сверки по камере |
| verified_by | FK → employee, NULL | Кто сверял |
| verified_at | TIMESTAMPTZ, NULL | Когда сверял |
| note | TEXT, NULL | Примечание |
| created_at / updated_at | TIMESTAMPTZ | Служебные метки |

**Ограничения:** частичный уникальный индекс `UNIQUE(employee_id) WHERE check_out_at IS NULL` (не более одного открытого визита). Индекс по `(employee_id, work_date)` для отчётов.

## 10. Нефункциональные требования
1. **Транспорт:** только HTTPS; валидные SSL-сертификаты; обязательная проверка подписи Telegram `initData`.
2. **Хранилище файлов:** реализуется через абстракцию для последующего перехода на S3-совместимое решение без изменения бизнес-логики.
3. **Целостность данных:** ключевые инварианты обеспечиваются на уровне БД (уникальные индексы, CHECK-ограничения, внешние ключи).
4. **Сопровождаемость:** единый переиспользуемый механизм для справочников; разделение бэкенда и фронтенда; миграции схемы через Alembic.
5. **Защита от перебора:** клиентские коды заказов — случайные и неугадываемые; ограничение частоты клиентских запросов.
6. **Масштаб:** решение рассчитано на одно небольшое предприятие; преждевременная оптимизация не требуется.
7. **Локализация:** двуязычный интерфейс (русский/узбекский) через файлы переводов; архитектура допускает добавление новых языков без изменения бизнес-логики.

## 11. Этапы разработки

Рекомендуемый порядок — от простого сквозного «скелета» к более сложным модулям.

1. **Основа и «тонкая вертикаль» на модуле посещений.** Простейшая функция (отметка прихода/ухода) проводится насквозь через весь стек: Telegram-авторизация → Mini App (Vue) → API (FastAPI) → PostgreSQL → отображение у руководителя. На этом этапе настраивается вся обвязка (модели, миграции, авторизация, хранилище).
2. **Модуль «Заказы».** Ядро системы: заказы, справочник этапов, прикрепление сотрудников и права, файлы, история статусов, клиентский просмотр через бота.
3. **Модуль «Финансы и отчётность».** Оплаты, расходы, справочник категорий, месячные отчёты. Идёт последним, так как опирается на данные заказов.

**Ближайший конкретный шаг:** модели SQLAlchemy и первая миграция Alembic.

## 12. Вне рамок текущей версии
Зафиксировано как сознательно невключённое (во избежание разночтений):
- перерывы/паузы внутри визита, графики работы и контроль опозданий;

- учёт отпусков и больничных;

- третий и последующие языки интерфейса (архитектура переводов это допускает).
