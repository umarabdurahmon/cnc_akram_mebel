## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (разделы 7.2 пп. 2–5, 9.6, и матрица 5.2).
Сначала план, потом код. Останавливайся на ревью перед миграцией.

### 1. Модель `order_employee` (тз 9.6, M2M)

`order_id` (FK → orders), `employee_id` (FK → employee), `can_change_status`
BOOLEAN, `attached_by` (FK → employee, null), `attached_at` TIMESTAMPTZ.
Ограничение: `UNIQUE(order_id, employee_id)`. Миграция — на ревью.

### 2. Прикрепление / открепление (manager)

- прикрепить сотрудника к заказу с флагом `can_change_status`;
- обновить флаг у существующего прикрепления (повторное прикрепление = upsert
  флага, не дубль — `UNIQUE` это и гарантирует);
- открепить — жёсткое удаление строки прикрепления безопасно: история статусов
  ссылается на сотрудника напрямую через `changed_by`, а не на прикрепление.

### 3. Объектное право на смену статуса (главный камень — IDOR)

Один централизованный чек `can_change_order_status(employee, order)`:
- `manager` → всегда `true`, прикрепление не требуется;
- `worker` → `true` ТОЛЬКО если есть строка `order_employee` для ЭТОГО
  `order_id`, этого `employee_id`, с `can_change_status = true`.

Обязательно:
- запрос фильтрует одновременно по `order_id` И `employee_id`. Право на заказ A
  не должно открывать заказ B (это и есть IDOR-дыра).
- чек в ОДНОМ месте (зависимость/сервис), не дублировать по эндпоинтам;
- никакого кеширования решения: отзыв `can_change_status` действует немедленно.

### 4. Эндпоинт смены статуса (расширяет Сессию 4)

- доступ — через `can_change_order_status`;
- при смене пишет `order_status_history` с `changed_by` = текущий сотрудник
  (рабочий тоже фиксируется в аудите);
- право УЗКОЕ: открывает только смену статуса. Редактирование полей заказа,
  файлы, финансы остаются `manager`-only — `can_change_status` их НЕ открывает.

### 5. Чтение заказов рабочим

По матрице 5.2 рабочий читает заказы (только чтение). По умолчанию — все заказы.
(Если захочешь ограничить только прикреплёнными — флагни в плане.)

### 6. Несущие тесты (только эти, настоящий Postgres, через auth-фикстуру)

- рабочий с `can_change_status` на заказе A меняет статус A → 200, история с
  `changed_by` = этот рабочий;
- ТОТ ЖЕ рабочий НЕ может сменить статус заказа B → 403 (ключевой IDOR-тест);
- рабочий прикреплён, но флаг `false` → 403; не прикреплён → 403;
- `manager` меняет статус любого заказа без прикрепления → 200;
- отзыв `can_change_status` немедленно даёт 403;
- `UNIQUE(order_id, employee_id)` соблюдается;
- прикрепление/открепление и редактирование полей доступны только `manager`.

### Definition of Done (Сессия 5)

- Руководитель прикрепляет сотрудников к заказу и точечно выдаёт право смены
  статуса; рабочий с правом меняет статус ТОЛЬКО своих заказов, история пишется
  с его авторством; отзыв права действует сразу; чужой заказ рабочему недоступен.
- `make migrate` чисто; `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.
