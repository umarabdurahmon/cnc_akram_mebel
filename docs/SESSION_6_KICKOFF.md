# Сессия 6 — Заказы: файлы (kickoff для Claude Code)

> Предусловие: Сессии 0–5 пройдены. Есть `orders`, прикрепления, права. Это
> первая работа с бинарными загрузками.
>
> Объём: модель `order_file`, загрузка по HTTPS, абстракция хранилища, миниатюры
> Pillow, валидация, список/скачивание/удаление. Клиентский просмотр — Сессия 7,
> Vue — 8. Файлы строго внутренние, клиенту НЕ отдаются (тз 8.5).

## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (раздел 8 целиком и 9.7). Сначала план, потом
код. Останавливайся на ревью перед миграцией.

### 1. Абстракция хранилища (тз 8.2, 10.2 — главный камень)

- Интерфейс `Storage` (Protocol/ABC): `save(key, data)`, `open(key) -> stream`,
  `delete(key)`. Бизнес-логика оперирует ТОЛЬКО непрозрачными `storage_key`
  (строками) — НИКОГДА путями ФС.
- `storage_key` генерит СЕРВЕР (uuid + расширение из ПРОВЕРЕННОГО типа), никогда
  из имени файла пользователя → это и закрывает path traversal.
- Локальная реализация мапит ключ в каталог (напр. `orders/{order_id}/...`),
  корень из `STORAGE_ROOT` (.env). Будущая S3-реализация мапит ключ в объект —
  бизнес-логику не трогаем.
- Скачивание — стримом через интерфейс, НЕ редирект на путь/статику ФС (иначе на
  S3 не переедет).

### 2. Модель `order_file` (тз 9.7)

`order_id` (FK), `original_filename` (для показа, НЕ для пути), `storage_key`,
`content_type` (проверенный), `size_bytes`, `kind` ENUM(`drawing`,`photo`,`other`),
`thumbnail_key` (null), `uploaded_by` (FK), `uploaded_at`. Миграция (ENUM) — ревью.

### 3. Загрузка (manager) и валидация

- multipart по HTTPS напрямую на бэкенд (НЕ через бота, тз 8.1);
- тип определять по СОДЕРЖИМОМУ (magic bytes), не по расширению / заявленному
  `content_type` (они от клиента);
- белый список типов (изображения, dwg/dxf, pdf и т.п.); лимит размера со
  стримингом (не грузить целиком в память);
- `original_filename` санитизировать для показа; для пути не использовать;
- `kind` — uploader указывает, дефолт из типа.

### 4. Миниатюры (Pillow, тз 8.3 / 8.4)

- только для изображений → `thumbnail_key`; CAD (dwg/dxf) и прочее → `thumbnail_key`
  null, на фронте иконка;
- guard декомпрессионной бомбы (`Image.MAX_IMAGE_PIXELS`), применить EXIF-ориентацию,
  маленький JPEG/WebP через тот же storage-интерфейс;
- генерацию миниатюры обернуть в try/except: битое изображение НЕ должно валить
  загрузку — файл сохраняем, миниатюру пропускаем.

### 5. Согласованность БД и хранилища (камень-бонус)

Две системы без общей транзакции:
- загрузка: сначала записать в storage (серверный ключ), потом коммит строки БД;
  при ошибке коммита — best-effort удалить записанный объект;
- удаление: сначала строку БД в транзакции, потом best-effort удалить объект
  (осиротевший файл менее вреден, чем висячая ссылка в БД).

### 6. Доступ (матрица 5.2)

Загрузка/удаление — `manager`. Просмотр/скачивание — и рабочий тоже.

### 7. Несущие тесты (только эти, настоящий Postgres + временное хранилище, auth-фикстура)

- загрузка изображения → строка + файл + миниатюра, `kind` корректный;
- загрузка dwg/dxf → файл сохранён, `thumbnail_key` null;
- `storage_key` серверный; имя файла с `../../` не выходит за пределы хранилища;
- превышение лимита размера → отказ; запрещённый тип → отказ;
- битое изображение → файл сохранён, миниатюра пропущена, без 500;
- загрузка/удаление только `manager`; рабочий может скачать;
- скачивание идёт через storage-интерфейс (не через путь ФС).

### Definition of Done (Сессия 6)

- Руководитель грузит файлы к заказу по HTTPS; изображения получают миниатюры,
  CAD — по иконке; типы и размер валидируются; хранилище за абстракцией;
  рабочий просматривает/скачивает; БД и хранилище не рассинхронизируются.
- `make migrate` чисто; `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.

Файлы строго внутренние — клиенту через бота не показываются (тз 8.5). Это
контракт для сериализатора Сессии 7.
