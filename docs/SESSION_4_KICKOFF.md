## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (разделы 7.2, 7.5, 9.3, 9.5, 9.8; и 6.2 про
код клиента). Сначала план, потом код. Останавливайся на ревью перед миграциями.
Все эндпоинты этой сессии — только роль `manager`.

### 1. Переиспользуемый справочник (тз 7.5, 10.4)

Сделай ОДИН generic-механизм «управление списком», который потом переиспользует
`expense_category` в финансах без переписывания:
- поля: `name`, `is_active`, и `position` (где нужен порядок);
- операции: список (активные/все), создать, переименовать, изменить порядок,
  деактивировать;
- удаление — ТОЛЬКО мягкое (`is_active = false`). Жёстко не удалять: на этапы
  ссылаются заказы и история статусов.

Первое применение — `production_stage` (`name`, `position`, `is_active`; тз 9.3).

### 2. Модель `orders` (тз 9.5)

Поля: `public_code` (UNIQUE), `internal_number` (null), `customer_name`,
`customer_contact` (null), `customer_chat_id` BIGINT (null, для будущих
уведомлений), `title`, `description` (null), `current_stage_id` (FK →
production_stage, null), `deadline` (null), `total_amount` NUMERIC(12,2) (null),
`public_note` (null), `created_by` (FK → employee), `created_at`/`updated_at`.
Миграция — на ревью.

### 3. Генерация `public_code` (камень)

- Криптостойкий RNG: модуль `secrets`, НЕ `random`.
- Алфавит без двусмысленных символов (без `0/O`, `1/I/l`), верхний регистр +
  цифры, длина ~6 (пример `K7QX2M`).
- Коллизии: `UNIQUE` на уровне БД + цикл «сгенерировал → вставил → при
  `IntegrityError` перегенерировал» (несколько попыток). НЕ пред-проверять
  существование (гонка) — БД арбитр уникальности.
- Клиенту НИКОГДА не отдавать `id` или `internal_number` (иначе защита
  `public_code` бессмысленна, тз 6.2). Это контракт для сериализатора Сессии 7.

### 4. CRUD заказов (manager)

create (генерит `public_code`, опционально стартовый этап), edit, list, get.

### 5. Смена этапа + история (тз 9.8)

- Смена `current_stage_id` ДОЛЖНА в одной транзакции писать
  `order_status_history` (`stage_id`, `changed_by`, `changed_at`, `comment`).
- Менять стадию без записи истории запрещено.
- Создание заказа со стартовым этапом — первая запись истории.

### 6. Несущие тесты (только эти, настоящий Postgres, через auth-фикстуру)

- `public_code` использует безопасный алфавит; при вставке дубля срабатывает
  `UNIQUE`, ретрай даёт другой код;
- деактивация этапа не удаляет строку; заказ, ссылающийся на этот этап, остаётся
  валидным; изменение порядка меняет `position`;
- смена этапа пишет ровно одну запись истории с правильным `changed_by`;
- CRUD заказов и справочника доступны только `manager`; рабочему → 403.

### Definition of Done (Сессия 4)

- Руководитель ведёт справочник этапов (мягкое удаление, порядок); создаёт и
  редактирует заказы; каждый заказ получает неугадываемый `public_code`; смена
  этапа фиксируется в истории.
- `make migrate` создаёт таблицы чисто и откатывается; `make check` зелёный.
- Закоммичено мелкими логическими коммитами.
