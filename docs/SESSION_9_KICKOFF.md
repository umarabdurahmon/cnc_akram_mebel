## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (разделы 7.4, 7.5, 9.4, 9.9, 9.10). Сначала
план, потом код. Ревью перед миграциями. Все эндпоинты — только роль `manager`.

### 1. Справочник категорий расходов (тз 9.4, 7.5)

Новый экземпляр переиспользуемого механизма из Сессии 4 (`name` + `is_active`,
порядок не нужен). Мягкое удаление; деактивированная категория не выбирается для
новых записей, но исторические расходы продолжают на неё ссылаться.

### 2. Деньги — точность (главный камень)

- все суммы `NUMERIC(12,2)` в БД, `decimal.Decimal` в Python; НИКОГДА не `float`
  — ни в арифметике, ни в сериализации;
- JSON: сериализовать `Decimal` как строку/точно, без float-коэрции (настроить
  Pydantic);
- агрегаты считать в SQL (`SUM`) — Postgres расширяет `NUMERIC`, переполнения нет;
  cap `12,2` касается только хранимых полей.

### 3. `order_payment` — оплаты (тз 9.10)

`amount` NUMERIC(12,2), `received_on` DATE, `note` (null), `created_by` (FK).
CRUD. manager.

### 4. `order_expense` — расходы, двухуровневая модель (тз 9.9)

- `direction` ENUM(`to_employee`, `general`);
- `category_id` FK → `expense_category`;
- `employee_id` (FK, null) — обязателен при `to_employee`;
- `amount` NUMERIC(12,2), `spent_on` DATE, `note` (null), `created_by` (FK).
- CHECK на уровне БД: `direction = to_employee ⇒ employee_id NOT NULL`.
  (Решить: запрещать ли `employee_id` при `general`.)
- Миграция (ENUM + CHECK) — на ревью.

### 5. Где ведутся

В карточке заказа руководителем (тз 7.4). Бэкенд-эндпоинты сейчас, UI — Сессия 11.

### 6. (Опционально) Остаток долга клиента

`total_amount − SUM(оплат)` — тривиально, можно отдать в карточке заказа
руководителю. Полный отчёт-дебиторка — вне v1 (тз 12), но данные собираются.

### 7. Несущие тесты (только эти, настоящий Postgres, через auth-фикстуру)

- расход `to_employee` без `employee_id` → отказ DB CHECK (и 400 в сервисе);
- расход `general` без сотрудника → ок;
- суммы остаются `Decimal`; сериализация и round-trip не теряют точность;
- оплата записывается; `SUM` оплат корректна; остаток = `total_amount − SUM`;
- категория из справочника; деактивированная не выбирается для новых, исторические
  расходы целы;
- оплаты/расходы доступны только `manager`; рабочему → 403.

### Definition of Done (Сессия 9)

- Руководитель ведёт по заказу оплаты и двухуровневые расходы (направление +
  категория); `to_employee` требует сотрудника на уровне БД; деньги считаются и
  сериализуются без потерь точности.
- `make migrate` чисто; `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.
