# Сессия 7 — Заказы: клиентский просмотр через бота (kickoff для Claude Code)

> Предусловие: Сессии 0–6 пройдены. Есть заказы, этапы, история, файлы. Это
> первый выход данных наружу, неаутентифицированному клиенту, через чат с ботом.
>
> Объём: безопасный сериализатор + логика бота (поиск по коду, deep-link, язык,
> rate limiting). Vue-интерфейс заказов — Сессия 8.

---

## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (разделы 7.3, 6.2, 7.6.4, 8.5). Сначала план,
потом код. Бот вызывает сервис in-process (общий слой `app/services`), без HTTP.

### 1. Безопасный сериализатор (главный камень — граница утечки)

- Отдельная функция/схема `serialize_for_client(order)`, собирающая результат из
  ЯВНО перечисленных безопасных полей (РАЗРЕШАЮЩИЙ список, не запрещающий):
  - `title`, `description`;
  - `deadline`;
  - `public_note`;
  - прогресс «этап X из Y» (см. п.2).
- НИКОГДА не отдавать: любые денежные суммы, файлы, прикреплённых сотрудников,
  внутренние комментарии, `id`, `internal_number` (тз 7.3, 8.5).
- Новое поле модели по умолчанию НЕ должно попадать к клиенту — только при явном
  добавлении в сериализатор (безопасность по умолчанию).
- Сериализатор — единственный источник истины: бот работает ТОЛЬКО с его
  результатом, в объект заказа сам не лезет.

### 2. Прогресс «этап X из Y»

- считать на лету по АКТИВНОМУ упорядоченному справочнику этапов:
  X — позиция текущего этапа, Y — число активных этапов;
- деактивированный этап не входит в Y;
- край: текущий этап заказа ссылается на ПОЗЖЕ деактивированный этап → обработать
  мягко (показать имя, не упасть, не делить на пусто);
- `current_stage_id = null` → осмысленный текст без прогресс-бара.

### 3. Бот: поиск по коду и deep-link

- клиент пишет код или переходит по `t.me/<bot>?start=<код>` (бот ловит payload
  в `/start` и сразу показывает статус);
- нормализация ввода: верхний регистр, обрезать пробелы (иначе валидный код не
  найдётся);
- код не найден → понятное «не найдено», без намёков, помогающих перебору.

### 4. Rate limiting (тз 6.2)

Лёгкое ограничение частоты поиска по коду (напр. throttling-middleware aiogram),
эшелон обороны от перебора.

### 5. Язык клиента (тз 7.6.4)

Определять по Telegram `language_code` либо предложить выбор при `/start`.
Переводятся ТОЛЬКО подписи/шаблоны; контент заказа и имена этапов — как ввёл
руководитель, не переводятся.

### 6. (Опционально) chat_id на будущее

Поле `customer_chat_id` уже есть в модели. Можно сохранять его при обращении
клиента (под будущие уведомления), но САМИ уведомления — НЕ в этой версии (тз 7.3.5).

### 7. Несущие тесты (только эти, настоящий Postgres, через сервис)

- сериализатор: заказ С деньгами, файлами, сотрудниками и внутренними заметками →
  в выводе НЕТ ни одного из этих значений/ключей (ключевой тест безопасности);
- «этап X из Y» считается по активным этапам; деактивированный не входит в Y;
- заказ с `current_stage_id = null` → без падения;
- заказ, чей текущий этап деактивирован → без падения;
- неизвестный код → «не найдено», без утечки;
- deep-link payload `/start=<код>` резолвится в статус;
- код находится независимо от регистра/пробелов;
- поиск по коду ограничен по частоте.

### Definition of Done (Сессия 7)

- Клиент по коду (вводом или deep-link) получает статус: название, описание,
  «этап X из Y», срок, публичный комментарий — и ничего сверх; неизвестный код
  даёт понятную ошибку; поиск ограничен по частоте; язык определяется.
- `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.
