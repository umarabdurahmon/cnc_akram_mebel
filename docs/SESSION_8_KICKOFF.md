# Сессия 8 — Заказы: Vue-интерфейс (kickoff для Claude Code)

> Предусловие: Сессии 0–7 пройдены. Бэкенд заказов готов целиком (CRUD, этапы,
> история, прикрепления+права, файлы, клиентский просмотр). Есть Vue-приложение,
> api-клиент, i18n и WebApp-интеграция из Сессии 3.
>
> Объём: интерфейс заказов для руководителя и рабочего. Завершает Этап 2 ТЗ.
> Дальше — Этап 3 (Финансы).

---

## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (разделы 7.2, 8, матрица 5.2). Сначала план,
потом код. Строй на фундаменте Сессии 3 (WebApp, api-клиент, i18n). Подписи
переводим (ru/uz); контент заказа, имена этапов, данные клиента — НЕ переводим.

### 1. Список и карточка заказа

- список заказов (manager — все; worker — чтение по матрице 5.2);
- карточка: поля заказа, текущий этап, история статусов, файлы, прикрепления
  (видимость секций — по роли).

### 2. Загрузка файлов из Mini App (камень 1)

- multipart по HTTPS прямо на бэкенд из вебвью (мимо Telegram, Сессия 6);
- api-клиент вешает `Authorization: tma <initData>` И на multipart (не только
  JSON); dev-мок initData (Сессия 3) распространяется на загрузку;
- `<input type="file">` в вебвью — учесть разницу пикера/доступа по платформам
  Telegram (iOS/Android/Desktop);
- прогресс загрузки + блокировка кнопки во время; лёгкая клиентская проверка
  размера/типа (accept) для быстрого отказа, но авторитет — сервер;
- отображение: миниатюры (`thumbnail_key`) для изображений, иконка для CAD/прочего;
  в списке НЕ грузить полноразмерные.

### 3. Интерфейс по правам (камень 2 — объектная авторизация Сессии 5)

- контрол смены статуса виден рабочему ТОЛЬКО если у него `can_change_status` на
  ЭТОМ заказе;
- фронт НЕ вычисляет право сам — бэкенд отдаёт в ответе по заказу вычисленный для
  текущего пользователя флаг (тем же централизованным чеком Сессии 5);
- скрытие кнопки = UX, НЕ безопасность: бэкенд всё равно отдаёт 403. Право отозвали
  между загрузкой и действием → показать 403 понятно, не падать;
- manager — полный набор (создание/правка/файлы/прикрепления/статус); worker —
  чтение + при праве смена статуса.

### 4. Роле-уместная проекция (камень-секундант)

- рабочий по матрице 5.2 НЕ видит финансы → бэкенд отдаёт рабочему проекцию заказа
  без финансовых данных;
- решить: видит ли рабочий `total_amount` (поле заказа, пограничное с финансами).
  Оплаты/расходы — Этап 3, их ещё нет.

### 5. Управление прикреплениями (manager)

Прикрепить/открепить рабочих, выставить `can_change_status` — UI поверх Сессии 5.

### 6. Смена статуса (manager и рабочий с правом)

Выбор этапа + комментарий → пишет историю (Сессии 4/5). После смены обновить
карточку и историю.

### 7. Проверка

- `make check` (eslint + сборка) зелёный;
- полная проверка вручную в Telegram через тоннель;
- (опц.) один компонентный тест: контрол статуса скрыт без права.

### Definition of Done (Сессия 8)

- Руководитель в Telegram ведёт заказы целиком: список, карточка, создание/правка,
  смена этапа с комментарием, загрузка файлов с миниатюрами, управление
  прикреплениями. Рабочий видит заказы (без финансов) и меняет статус только там,
  где есть право; скрытие контролов — UX, защищает бэкенд. Загрузка из вебвью
  работает на целевых платформах.
- `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.

Это завершает Этап 2 ТЗ (заказы: бэкенд + бот + Vue). Дальше — Этап 3: Финансы
(оплаты, расходы по заказам, операционные расходы, месячные отчёты).
