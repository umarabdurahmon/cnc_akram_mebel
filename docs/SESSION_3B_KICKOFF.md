# Сессия 3b — Vue Mini App: сверка и ручная правка (kickoff для Claude Code)

> Предусловие: Сессии 0–3 пройдены. Бэкенд 2b готов (verify, discrepancies,
> PATCH-правка с провенансом). Есть Vue-приложение, api-клиент, i18n и сводка
> руководителя из Сессии 3.
> Это UI руководителя поверх готового бэкенда. Завершает модуль посещений.

---

## Промпт

Прочитай `CLAUDE.md` и `docs/тз.md` (раздел 7.1, пункты 7 и 8). Сначала план,
потом код. Весь интерфейс этой сессии — только для роли `manager` (бэкенд это
тоже проверяет; клиентская проверка — лишь для отрисовки).

### 1. Сверка — в строке сводки, одним тапом

В сводке руководителя из Сессии 3 на каждой записи:
- видимый текущий `verification_status` (иконка/цвет: `not_checked` /
  `confirmed` / `discrepancy`);
- действие в один тап: пометить `confirmed` или `discrepancy` →
  `POST /api/attendance/{id}/verify`;
- после ответа показать, кто и когда сверял (`verified_by` / `verified_at`).

### 2. Ручная правка — форма (главный камень: время по Ташкенту)

Правка `check_in_at`, `check_out_at`, `note` (типично — закрыть «забытый» визит).
- Поля времени работают и подписаны явно как «время по Ташкенту».
- Узбекистан — фиксированный UTC+5 без летнего времени: конвертация
  введённого настенного времени в UTC — постоянный сдвиг, но делать её НАДО
  явно, не полагаясь на часовой пояс устройства / на `datetime-local` вслепую.
- После правки `check_in_at` показать пересчитанный `work_date` (бэкенд меняет
  его в `Asia/Tashkent`), чтобы перенос через полночь был виден.
- Показать «было → стало» по часам за день.

### 3. Подтверждение последствий — нативные диалоги Telegram

- Правка меняет учтённые часы (→ подотчёт в финансах), поэтому перед сохранением
  спрашивай подтверждение через `WebApp.showConfirm` / `showPopup`, с показом
  «было → стало». НЕ используй браузерный `alert()`.
- Ошибки и уведомления — тоже через нативные диалоги.

### 4. Ошибки бэкенда — без оптимистичного коммита

- 400 (`check_out_at < check_in_at`) и 409 (reopen при уже открытом визите) →
  понятные локализованные сообщения через нативный диалог.
- НЕ показывай правку как применённую до ответа сервера; при ошибке откатывай
  форму к исходному состоянию.

### 5. Видимость провенанса

На отредактированной записи показывай «изменено: <кто> в <HH:MM по Ташкенту>»
(поля провенанса, выбранные в 2b). Провенанс должен быть виден, иначе он бесполезен.

### 6. (Опционально) Статистика расхождений

Небольшой блок/экран: `GET /api/attendance/discrepancies?from=...&to=...` —
по сотрудникам за период. Только manager.

### 7. Локализация

Статусы (`confirmed`/`discrepancy`/`not_checked`) и все сообщения — переводим
(ru/uz). Имена сотрудников и заметки — пользовательский контент, НЕ переводим.

### 8. Проверка

- `make check` (eslint + сборка) зелёный.
- Полная проверка вручную в Telegram через тоннель.

### Definition of Done (Сессия 3b)

- Руководитель в Telegram спот-чекает записи одним тапом (статус виден); правит
  «забытые» визиты с явным временем по Ташкенту и подтверждением; видит провенанс
  и пересчитанные часы; ошибки 400/409 показываются понятно нативным диалогом.
- `make check` зелёный, показан вывод.
- Закоммичено мелкими логическими коммитами.
