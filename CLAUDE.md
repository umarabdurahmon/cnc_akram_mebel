# Furniture System (CNC mebel bot)

Кастомная система для одного небольшого мебельного предприятия. Работает
целиком в Telegram: бот (aiogram) + Mini App (Vue) поверх единого бэкенда
(FastAPI / PostgreSQL). Три модуля: посещения, заказы, финансы.

Полное техническое задание — `docs/тз.md`. Это источник истины по требованиям.
Цветовая система фронтенда — `docs/color-system.md`. Все CSS-токены и правила их применения.
Навигация по коду (где что живёт) — `docs/nav.md`.

## Стек

- Python 3.12+, FastAPI, SQLAlchemy 2.x + Alembic, aiogram 3.x, Pillow
- Vue 3 + Vite (Mini App)
- PostgreSQL 16
- Тесты/качество: pytest, ruff, mypy

Точные версии зафиксированы в `server/pyproject.toml` и `frontend/package.json`.

## Архитектурные инварианты (НЕ нарушать)

- Бот и API делят слой `server/app/services`. Бот вызывает сервисы напрямую
  (in-process), Mini App ходит к API по HTTPS. Не дублируй логику между ними.
- Авторизация сотрудников — ТОЛЬКО через Telegram `initData` (проверка HMAC по
  токену бота). Паролей и логинов нет. Личность = `telegram_id`.
- Клиент — без авторизации, доступ к статусу заказа по случайному `public_code`
  (неугадываемый код, не порядковый номер).
- Клиентский сериализатор НИКОГДА не отдаёт: деньги, файлы, прикреплённых
  сотрудников, внутренние комментарии. Только безопасное подмножество (см. тз 7.3).
- Деньги: тип `NUMERIC(12,2)`, валюта единая (UZS).
- Инварианты целостности живут на уровне БД (частичные уникальные индексы,
  CHECK, внешние ключи), а не только в коде Python.
- Файловое хранилище — через абстракцию (интерфейс), под будущий переход на S3.
  Не привязывайся к локальной ФС напрямую в бизнес-логике.
- Справочники (этапы, категории расходов) — мягкое удаление через `is_active`.
  Жёстко не удалять, чтобы не разрушать историю.
- Цвета во фронтенде — ТОЛЬКО через CSS-переменные из `docs/color-system.md`.
  Никаких hex/rgb/rgba напрямую в `<style scoped>`. Новый семантический цвет →
  добавить токен в `:root` в `App.vue` и задокументировать в `docs/color-system.md`.
- Переводится ТОЛЬКО интерфейс (vue-i18n + i18n aiogram). Значения справочников
  и пользовательский контент (названия/описания заказов, комментарии) — НЕ
  переводятся, показываются как ввёл руководитель.

## Команды

- Поднять Postgres: `make up`
- Полная проверка: `make check` (ruff + mypy + pytest + сборка фронта)
- Только тесты: `make test`
- Линт + типы: `make lint`
- Применить миграции: `make migrate`
- Новая миграция: `make revision m="описание"`
- Запуск процессов: `make dev-api`, `make dev-bot`, `make dev-front`

## Тестирование

- Тесты пишем ТОЛЬКО на несущие инварианты, не ради покрытия. Список несущих:
  клиентский сериализатор не протекает; нельзя два открытых визита; подсчёт
  часов; CHECK по расходам (`to_employee` требует сотрудника); проверка
  `initData`; право рабочего на смену статуса только по выданному заказу.
- Тесты бьют в НАСТОЯЩИЙ Postgres (база `furniture_test`), а не в SQLite —
  иначе БД-инварианты не проверяются. Каждый тест — в транзакции с откатом.

## Правила работы

- Перед нетривиальной задачей — сначала план (plan mode), потом код.
- Любую миграцию Alembic показывай на ревью ДО применения.
- Не утверждай «готово» без вывода `make check`. Показывай реальный вывод
  команды, а не его пересказ.
- Коммит — на каждом зелёном чекпоинте, мелкими логическими единицами.
- Каждый коммит должен оставлять `make check` зелёным.

## Порядок разработки

Идём «тонкой вертикалью», см. `docs/тз.md` раздел 11:
1. Каркас + посещения (шагающий скелет насквозь через весь стек).
2. Заказы (ядро: справочник этапов, права, файлы, история, клиентский просмотр).
3. Финансы и отчётность (опирается на данные заказов, идёт последним).
